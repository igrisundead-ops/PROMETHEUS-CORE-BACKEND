вңҰ Here is the DEEP SHERLOCK report. I have
  interrogated the codebase across
  remotion-app/src/lib/cinematic-typography,
  web-preview/NativePreviewStage.tsx,
  typography-intelligence.ts, and the font loading
  pipeline.

  The preview does not look premium because the
  system is secretly sabotaging itself at the
  rendering, CSS, and graph layers.

  1. FONT REALITY CHECK
  Hidden Failure: "Faux Bold" Synthesis.
  The system requests fonts via
  @remotion/google-fonts using
  loadFraunces("normal", ...) and
  loadDMSans("normal", ...) without explicitly
  loading specific numeric weights. However,
  resolveCaptionEditorialDecision routinely
  assigns fontWeight: 800 or fontWeight: 700 to
  the DOM. Because the browser does not have the
  800-weight file loaded, Chrome executes
  synthetic bolding (faux bold). This destroys the
  font's designed kerning, smears the letterforms,
  and creates a distinctly "cheap" appearance that
  ruins premium serif and sans-serif fonts alike.

  2. FONT QUALITY CHECK
  Hidden Failure: Hardcoded Generic Safety.
  The graph attempts to calculate a
  readabilityScore, but in
  runtime-font-selector.ts, there is a hardcoded
  massive bias: if (candidate.id === "dm-sans" &&
  roleId === "neutral_sans_core") { score += 0.95;
  }. This override functionally guarantees that DM
  Sans wins the majority of the time, making the
  "elite" font intelligence system output a
  generic tech-startup font even when a luxury
  serif would be mathematically better.

  3. FONT COMPATIBILITY CHECK
  Hidden Failure: Bypassed Edge Scores.
  The system has a rich FONT_COMPATIBILITY_EDGES
  graph with relations like supports, contrast,
  and avoid. But the execution layer
  (resolveOperationalRoleId) falls back to
  neutral_sans_core for nearly all edge cases. The
  graph theoretically understands hierarchy, but
  the execution forces the safest,
  lowest-common-denominator font.

  4. CONTENT/TEXT CHECK
  Hidden Failure: Naive Emphasis Guessing.
  The system doesn't truly understand cinematic
  pacing blocks (Hook, Context, Proof, CTA). It
  relies on primitive regex (/\?$/ implies "hook")
  and counts (emphasisCount >= 2). It treats all
  lines exactly the same geometrically, applying
  identical motion to a 9-word explanatory
  sentence and a 2-word hook.

  5. LAYOUT CHECK
  Hidden Failure: Cramped Horizon Sprawl.
  In preview-subtitle-system.ts, maxWidthPercent
  is allowed to reach up to 80%. Cinematic
  typography demands constraint. Pushing text to
  80% of the screen width forces the viewer's eyes
  to ping-pong across the frame, destroying the
  focused, "locked" cinematic feeling. It is
  mathematically centered rather than
  rule-of-thirds weighted.

  6. CSS CHECK
  Hidden Failure: Muddy "Social Template" Glows.
  The system hardcodes massive, overlapping
  shadows that kill premium aesthetics. In
  NativePreviewStage.tsx:
  textShadow: 0 8px 22px rgba(0,0,0,0.58), 0 16px
  44px rgba(0,0,0,0.2).
  Furthermore, an explicit multiplier
  PREVIEW_SUBTITLE_GLOW_GAIN pushes glows up to
  34px. Elite typography uses minimal, razor-sharp
  drop shadows (e.g., 0 2px 4px), not massive
  muddy glows that make text look like a 2018
  YouTube thumbnail.

  7. ANIMATION CHECK
  Hidden Failure: React Render Loop Thrashing.
  The motion is driven by JS easeOutCubic
  interpolating currentTimeMs inside React render
  cycles (NativePreviewStage.tsx). For a 12-word
  sentence, this forces React to recalculate and
  commit 12 separate transform and filter styles
  to the DOM every 16ms. This guarantees
  micro-stutters and dropped frames, completely
  breaking the "buttery smooth" premium feel.

  8. VIDEO CONTEXT CHECK
  Hidden Failure: Metadata Illusion.
  The system derives CaptionSurfaceTone
  (light/dark/neutral) strictly from a static
  gradeProfile.brightness value for the entire
  clip. It does not sample the actual pixels
  behind the text dynamically. If a video is
  globally "dark" but has a bright window exactly
  where the text sits, the text will have the
  wrong contrast treatment and become unreadable.

  9. RENDERING/PREVIEW CHECK
  Hidden Failure: The Subpixel Anti-Aliasing
  Killer.
  Every animated letter/word receives will-change:
  transform, opacity, filter and filter:
  blur(...). When Chrome sees this, it promotes
  the span to an accelerated GPU texture. Chrome
  completely disables subpixel anti-aliasing for
  GPU-composited text layers. The text is rendered
  with standard grayscale anti-aliasing, making it
  look instantly thin, jagged, and heavily
  degraded compared to standard DOM text.

  10. ELITE STYLE CHECK
  Comparing this to Dean Graziosi or Iman Gadzhi:
   - Weights: Elite styles use properly loaded
     Black/Ultra-Bold variants. We use faux bold.
   - Spacing: Elite styles use tight, negative
     tracking on bold uppercase (-0.04em). We use
     positive tracking (0.024em) on uppercase,
     causing it to scatter.
   - Motion: Elite styles use sharp, sub-frame
     scale locks. We use slow, 800ms blurred
     drifts that linger in a blurry state.

  11. SCORING CHECK
  There is no final unified render score before
  commitment. If there were, here is the current
  diagnostic estimate:
   - fontLoadScore: 0.2 (Faux bold synthesis
     detected)
   - fontPremiumScore: 0.4 (DM Sans override
     forced)
   - readabilityScore: 0.5 (Muddy 34px glow behind
     text)
   - animationTasteScore: 0.3 (Blur lingers too
     long, React stutters)
   - renderReliabilityScore: 0.1 (Subpixel
     anti-aliasing killed by GPU promotion)

  12. CANDIDATE CHECK
  The system is highly fragile because it
  generates exactly one CaptionEditorialDecision.
  It does not pit 5 candidates (Safe Premium,
  Aggressive Hook, Editorial, SaaS, Caption-first)
  against each other. If the weak regex guesses
  the mood is "tech", the entire scene is locked
  into a generic tech aesthetic with zero
  fallback.

  13. NEGATIVE GRAMMAR CHECK
  Missing Bans:
  typographyAvoidOveruse is just a list of English
  strings ("too much blur on all captions"). It is
  not compiled into deterministic runtime bans.
  Missing deterministic bans:
   - Banned: will-change on text without
     backface-visibility: hidden and transform:
     translateZ(0) compensation.
   - Banned: Positive letter-spacing on
     heavyweight fonts.
   - Banned: Shadows exceeding 12px blur radius on
     standard premium roles.
   - Banned: maxWidthPercent > 65%.

  Top 5 Hidden Quality Killers
   1. Faux Bold Synthesis: loadFraunces("normal")
      without numeric weights forces the browser
      to fake fontWeight: 800.
   2. Subpixel AA Death: filter: blur() combined
      with will-change turns off subpixel
      anti-aliasing in Chrome, making premium
      fonts look jagged.
   3. The DM-Sans Hardcode: score += 0.95
      completely overrides the intelligence graph.
   4. Muddy Glow Injection:
      PREVIEW_SUBTITLE_GLOW_GAIN scales generic
      shadows up to 34px.
   5. React Main-Thread Motion: Recalculating 12
      transforms per frame in React causes jitter.

  Top 5 Fastest Fixes
   1. CSS Fix: Update the textShadow defaults to
      tight, cinematic specs (e.g., 0 2px 6px
      rgba(0,0,0,0.8)).
   2. Layout Fix: Hard-cap maxWidthPercent to 65
      in preview-subtitle-system.ts.
   3. Data Quality Fix: Remove the +0.95 DM Sans
      override in runtime-font-selector.ts.
   4. Rendering Fix: Add transform: translateZ(0)
      or remove will-change: filter at the end of
      the animation settle phase to restore
      Subpixel AA when the text is resting.
   5. Font Loading Fix: Explicitly load specific
      weights (600, 700, 800) in
      house-font-loader.ts to kill faux-bolding.

  Top 3 Architectural Upgrades (Do Not Build Yet)
   1. Candidate Pipeline: Shift from
      single-decision generation to a 5-candidate
      ranking system.
   2. Canvas/Frame Sampling: Read actual
      background pixel brightness to drive
      surfaceTone instead of relying on global
      metadata.
